## From the RM-06 independent security review (2026-07-07)
The review's MUST-FIX (#1 DEFINER search_path + revoke CREATE; #2 consent-gate all raw reads) and SHOULD-FIX (#3 unforgeable audit; #4 no consent oracle; #5 FORCE RLS; #6 assignments provenance) were fixed in migration `0006_hardening.sql`. The following were consciously deferred:

- **SEC-REV-07 — Legacy-PIN name lockout is a targeted DoS.** `record_attempts` (0001) locks a known player name for 15 min after 5 bad PINs, repeatable. *Deferred because* name+PIN auth is being retired for real Google OAuth (SPEC RM-12) and the legacy game rows are frozen (RM-15). Revisit only if PIN auth outlives the cutover; then IP-scope the throttle / degrade to CAPTCHA. Severity: low.
- **SEC-REV-08 — Direct `child_context_pack` / `authorize` callers bypass the gateway's verify/moderate/audit.** An authenticated user can call these RPCs directly, skipping the Edge Function's choke points. *Deferred because* the pack is a name-free whitelist of data the caller is already authorized to read (same as `mastery_select`), and verify/moderate only guard generated text the RPC never produces; the only loss is audit-completeness for direct reads (never audited anyway). If complete AI-access auditing becomes required, route pack access behind one audited RPC and revoke direct `execute` of `child_context_pack` from `authenticated`. Severity: low. (Relates to KER-1/KER-6.)
- **SEC-REV-09 — `teaching_artifacts` INSERT does not constrain `visibility_scope`.** A writer may set any scope on their own artifact for a child they write to. *Deferred because* no cross-family leak results — `teaching_artifacts_select` still gates every read by `can_view_child` + consent; this is authorization hygiene only. Constrain settable scope per `author_role` if scopes gain behavioral meaning. Severity: low. (Relates to DM-15.)
- **SEC-REV-10 — Edge Function CORS `Access-Control-Allow-Origin: *`.** *Cleared as acceptable* — the function is bearer-token (JWT) authorized, not cookie-based, so `*` does not enable credential replay. Recorded for completeness. Severity: informational. (Relates to SEC-01.)

## From RM-07 (Group + Membership derivation engine) design (2026-07-07)
- **DM-11 — full event-log consolidation, deferred.** SPEC DM-11 calls for exactly ONE append-only Event log with a `kind` discriminator. RM-07 introduces one new unified `events` table for the derivation-era kinds (membership, requirement, attendance, completion, signature, message, flag, payment_intent, schedule, derivation_audit), but keeps `attempts`, `consent_ledger`, `audit_log`, and `teaching_artifacts` as their existing specialized append-only logs. *Deferred because* folding `attempts` into a generic log would break the proven capture spine, mastery projection, and reconciler (RM-01) for no functional gain. All four remaining logs are append-only + immutable, so DM-11's invariant holds in spirit. Revisit by first-principles review only if the split ever bites (e.g. a cross-kind query need).
- **DM-1 — unified `actors` table, deferred.** SPEC DM-1 calls for every person/agent to be one row in a single `actors` table. RM-07 keeps the current model — adults are `auth.users`, child actors are `children` — and `memberships` reference either via nullable `member_actor_id` / `member_child_id` + an exactly-one CHECK. *Deferred because* a unified actors table would rebuild working auth/RLS/grants that all key on `auth.uid()` and `children`, with no functional gain now. Revisit by first-principles review if a case needs adults and children in one id space (e.g. an adult who is also a "learner").

## From the 0007–0010 red-team review (2026-07-07)
The review's MUST-FIX (C1 group-join ownership, C2 child-subject event isolation, H3 solver recompute) and MEDIUM (M4 drain worker-only, M5 approval idempotency, M6 rebuild consent, M7 delivery moderation) + L9/L10 were fixed in `0011_review_hardening.sql`. Deferred:

- **SEC-REV-M8 — `propose_grade`/`propose_assignment` trust client-supplied `model`/`prompt_version`.** These DEFINER writers stamp `author_role='ai'` server-side but accept the AI provenance fields from the caller, so a parent can forge which "model/prompt version" graded their own child (provenance/audit pollution; **own-family only, and the verdict is still recomputed by the solver**, so no cross-family or correctness impact). *Deferred because* the real fix is a distinct **AI-worker service identity** for the gateway (SPEC ROLE-8 / AI-1: the AI is an Actor holding a Grant with an `owner_actor_id`) — the `propose_*` writers should then be gated to that identity, and `model`/`prompt_version` derived server-side. Until then these fields are self-asserted provenance, not trustworthy. Revisit when the AI-worker actor lands. (SEC-07 / KER-6.)
- **SEC-REV-L11 — `post_message` accepts an arbitrary `context_ref` and message events are group-wide, not per-channel.** A member can weld a message to any context ref, and message events (group-scoped, no `subject_child_id`) are readable by every `is_group_member`, not only members of the specific channel. *Deferred because* it is acceptable for a shared group feed and no cross-family leak results (the group branch is family-isolated). Revisit when channel-level isolation / per-channel membership visibility is built (COM-02/05). Severity: low.

- **AUDIT-HARDENING (stronger option, deferred).** `0006` made audit writes unforgeable via decision-recompute + detail cap + definer source stamp (finding #3, minimal). The larger restructure — revoke `write_audit` from `authenticated` and route the whole spine through one audited RPC so clients can only trigger the full path, never a bare audit write — remains available if evidentiary requirements tighten. (SEC-07 / ACC-06.)
